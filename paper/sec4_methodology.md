## 4.1  Workload

We evaluate on **LC_VCO_20GHz**, a 20 GHz inductively-loaded
differential VCO laid out on a real industrial-node CMOS PDK
(specific node withheld under NDA; reported in supplementary
material with metric scales normalised to preserve compliance).
The boilerplate scope statement we carry across §1, §4, and the
abstract: *we evaluate on LC_VCO_20GHz; an additional OpAmp
testbench will be reported in revision/appendix if available before
submission deadline.* The single-circuit scope is the honest extent
of the D8 deliverable; the OpAmp hook is intentional but not
committed. The cobi_matching second case study previously
considered in v2 is **dropped** per user D1-evening pivot — this
benchmark is sizing-style only, not Ising / combinatorial.

## 4.2  Spec contract

The spec is a YAML metric contract (signals / windows / metrics)
identical across all eleven LLMs. The seven concrete metrics for
LC_VCO_20G — taken directly from the live spec
(`projects/lc_vco_base/constraints/spec.md:61-88`) — are: `f_osc_GHz`
(centre frequency, derived from `freq_Hz` on `Vdiff`),
`V_diff_pp_V` (late-window peak-to-peak swing), `V_cm_V` (late-
window common-mode mean), `duty_cycle_pct`, `amp_hold_ratio`
(late/early RMS ratio characterising oscillation hold-up),
`t_startup_ns` (time to first crossing 45 % of the late-window
swing), and `I_core_uA` (mean tail current). The evaluator supports
only the simple stats `{mean, min, max, ptp, rms, mean_abs,
freq_Hz, duty_pct}` plus the compound forms `{ratio, t_cross_frac}`
(`src/spec_evaluator.py:104-106`); phase-noise / K_VCO are out of
scope for v3. A run **passes** iff every metric is within its
`pass` range; failures classify by which metric fell out of range
(`MEASURED_OUT_OF_RANGE`) versus the evaluator declaring the
measurement infeasible (`UNMEASURABLE`).

## 4.3  Protocol

For each (LLM, run-seed) tuple the agent runs the same Safe-Bridge
configuration, the same prompt template, and the same JSON tool
contract; only the `--llm` flag differs across cells of the matrix.
Each run iterates until either (i) the spec evaluator reports first-
pass, or (ii) a stuck-streak guard (no design-vector change across
3 successive iterations) aborts, or (iii) an iteration budget cap
of N (resolved per pilot) is reached. The eleven checkpoints in
scope:

| Vendor    | Country | Checkpoint        | Reasoning channel             |
|-----------|---------|-------------------|-------------------------------|
| Anthropic | US      | Opus 4.7          | thinking tokens               |
| Anthropic | US      | Sonnet 4.6        | thinking tokens               |
| Anthropic | US      | Haiku 4.5         | thinking tokens               |
| OpenAI    | US      | GPT-5.5           | `reasoning_tokens`            |
| OpenAI    | US      | GPT-5.4-mini      | `reasoning_tokens`            |
| Google    | US      | Gemini 2.x        | `thinking` block (TBD-final)  |
| Moonshot  | China   | Kimi K2.5         | `reasoning_content`           |

## 4.4  Seed handling and labelling

To pre-empt the most common reviewer objection on multi-LLM
benchmarks — that LLMs see different prompts because the seed
shapes the spec — we explicitly disclose: **seed = label**. Each
run is keyed by a value from the fixed replicate set `{1, 2, 3}`
(`scripts/run_benchmark.py:94`); the same three keys are run for
every LLM in the matrix. The spec contract is *identical* across
LLMs (byte-for-byte), and `build_subprocess_cmd`
(`scripts/run_benchmark.py:187-217`) does not pass the seed to
`run_agent.py` or to any API request — it is used only for
per-replicate result-directory naming and for downstream
aggregation. The LLM therefore cannot differentiate its runs by
seed identity: the seed is a pure replication label, not a prompt
or sampling-API input.

## 4.5  Repetition and variance

Each LLM is run for at least **three independent seeds** per cell of
the matrix (more for LLMs identified as noisy in pilot). Reported
quantities are **median + interquartile range (IQR)**, not best-of;
single-run reports are flagged as such in §5. The 11 × ≥3 floor is
33 runs minimum; with the failure-probe spec for §6 contributing 11
× ≥5 = 55 more, the D8 budget is ~88+ full sizing-loop runs.

## 4.6  Three-axis scrub-audit methodology

The trust-boundary defences in §3 are only as strong as the test
fixtures that verify them. We surface a paper-level methodology
that other closed-loop EDA agents should adopt: every scrub-coverage
regression test asserts on three independent axes simultaneously:
(a) **records-present** — the expected log record is emitted at the
expected level; (b) **forbidden-tokens-absent** — every member of a
known PDK-token set is absent from the emitted text; (c) **scrub-
marker-present** — the redaction marker (`<redacted>` / `<path>`)
is present, confirming the scrubber ran rather than the leak path
being unreachable in the test fixture. A single negative assertion
("the forbidden token is absent") gives a false positive when the
log line itself is never emitted; the three-axis pattern catches
that failure mode. The canonical implementation is
`tests/test_llm_client.py::TestOllamaThinkingFallback::test_debug_log_scrubs_thinking_pdk_tokens`
(`tests/test_llm_client.py:147-190`), embodied at the reasoning-
trace scrub call sites in `llm_client.py` (Kimi `:339`, MiniMax
`:431`, OpenAI `:533`, MiMo `:636`, DeepSeek `:742`, Ollama debug
`:810`, Ollama fallback `:818`).

## 4.7  Metric definitions (Table 2)

| Metric                | Definition                                                                                         |
|-----------------------|----------------------------------------------------------------------------------------------------|
| **Success rate**      | fraction of runs reaching all-metric-pass (∈ [0, 1]); reported as median across seeds              |
| **Iter-to-converge**  | iteration count of first all-metric-pass (∞ if never)                                              |
| **Input tokens**      | sum across all turns of a run; per-vendor token counter as documented in each provider's SDK       |
| **Output tokens**     | sum across turns; for OpenAI excludes `reasoning_tokens`; for Kimi / MiniMax / MiMo / DeepSeek excludes `reasoning_content` (Anthropic / Gemini accounting is input + output + total only in v3 — see §5.5 caveat) |
| **Reasoning tokens**  | broken out separately for vendors whose API surfaces them on the current `_normalize_usage` schema (`src/llm_client.py:21-80`); Anthropic / Gemini reasoning-token usage is **not currently tracked** and is flagged as a v3 limitation in §5.5 / §8.2 |
| **USD per run**       | tokens × public pricing as of **2026-05-01**; pricing table in appendix; reasoning tokens billed at the documented per-vendor rate for vendors with tracked reasoning-token counts |
| **Wall-clock / iter** | median over completed iterations; excludes rate-limit / 429 retry waits, separately reported       |

## 4.8  Threats to validity

We enumerate the eight validity threats reviewers most often raise
on benchmark papers, each with our intended response in §5/§6/§7:

1. **Single-circuit scope.** Generalisation to other analog
   topologies is unproven. We state this up-front and use the
   "OpAmp in revision/appendix" hook (§4.1) to flag the open
   problem without promising a result we have not yet measured.
2. **Pricing drift.** Public per-token pricing changes; we pin all
   USD figures to a single 2026-05-01 pricing snapshot, with the
   raw token counts also reported so re-pricing under future
   tariffs is straightforward.
3. **Rate-limit / 429 noise.** Inflates wall-clock for cost-floor
   LLMs (Haiku 4.5, GPT-5.4-mini, DeepSeek V4-flash, MiMo v2.5-pro)
   asymmetrically. We separate rate-limit wait time from compute
   wall-clock and report both.
4. **Reasoning-token accounting per-vendor differences.** Kimi,
   MiniMax, OpenAI, MiMo, and DeepSeek all return an OpenAI-
   compatible usage shape; `_normalize_usage` reads the
   `completion_tokens_details.reasoning_tokens` sub-field and
   surfaces it as the dedicated `reasoning_tokens` counter
   (`src/llm_client.py:47-59`). The associated `reasoning_content`
   *text* (assistant reasoning trace, distinct from the usage
   counter) is the scrubbed-fallback path handled separately at the
   per-vendor reply sites (Kimi `:339`, MiniMax `:431`, OpenAI
   `:533`, MiMo `:636`, DeepSeek `:742`). Anthropic's thinking-
   token accounting (`cache_creation_input_tokens` /
   `cache_read_input_tokens`) and Gemini's `thinking`-block usage
   are **not currently captured** by `_normalize_usage`
   (`src/llm_client.py:42-46`); the Anthropic and Gemini rows of
   the §5.5 premium table are therefore marked unavailable, with a
   schema extension flagged in §8.2.
5. **Provider-side caching.** Reduces reported input-token cost in
   the latter half of a run for vendors with cache. Cache-adjusted
   reporting requires capturing the `cache_creation` / `cache_read`
   split, which is **not in scope** under the current schema (see
   threat 4); we report raw token counts only in v3.
6. **Cohort selection bias.** Doubao-pro / GLM-5 / Qwen3-Max are
   not in the eleven-checkpoint cohort; the selection criterion was
   *vendors with stable public reasoning-mode APIs as of 2026-05-12
   that fit the cloud-LLM-with-filter framing*. On-prem-only
   checkpoints are out of scope (orthogonally covered by AnalogAgent
   [Bao et al., 2026]).
7. **Prompt-template optimality.** A single template is used across
   all eleven LLMs. This is the fair-comparison axis by construction
   but is not vendor-optimal for any individual checkpoint; per-LLM
   prompt tuning would be a follow-on study.
8. **Sampling-parameter standardisation.** Vendor defaults for
   temperature / top-p / top-k differ; some reasoning LLMs (OpenAI
   o-series) ignore temperature entirely. We use each vendor's
   default sampling parameters for the headline result and report
   the configuration per LLM in the supplementary appendix.
