## 3.1  Trust boundary and threat actor

The benchmark runs across eleven cloud LLM endpoints drawn from a
set of seven commercial providers (Anthropic ×3, OpenAI ×2, Google
×1, Moonshot ×1, MiniMax ×1, Xiaomi ×1, DeepSeek ×2). Following the v2 threat-model
framing, we treat every such provider as **curious-but-passive**: it
sees every prompt and every tool-result string we hand it, may log
or train on that traffic, and can craft tool-call arguments designed
to exfiltrate foundry content via its own sandbox — but is not
assumed to mount active man-in-the-middle attacks against the
Cadence host. Active MITM, OS-level insider threat, and direct PDK
theft are out of scope. The protected asset is foundry NDA-bound
PDK content in three forms: BSIM4 / BSIM-CMG model parameters,
process-stack layer data, and proprietary cell topology and naming.

## 3.3  Four scrub layers

Safe-Bridge inserts four defense layers, each labelled on Fig. 1:

1. **Python parameter-name whitelist + identifier / numeric-literal
   regex.** `_is_allowed_param_name` (`safe_bridge.py:2310-2356`)
   admits only the per-project YAML allow-list (defaulted to
   `[w, l, nf, m, multi, wf, r, c, nfin, fingers, idc, vdc]`),
   passing `_SAFE_PARAM_NAME_RE` (`safe_bridge.py:186`) and never
   `_FOUNDRY_LEAK_RE`. A complementary numeric-atom regex
   (`safe_bridge.py:35-37`) blocks SKILL injection at the value
   boundary.
2. **20-name SKILL entrypoint allow-list (Cadence side).**
   `_ALLOWED_SKILL_ENTRYPOINTS` (`safe_bridge.py:125-176`); arbitrary
   `load() / hiOpenLib / dbOpenCellViewByType` rejected at
   `_check_skill_entrypoint` (`safe_bridge.py:2149-2212`). Nested-call
   grep extends the guarantee to embedded subexpressions; only the
   pure data-constructor `list` is whitelisted as a nested call
   (`_ALLOWED_SKILL_NESTED`, `safe_bridge.py:181`).
3. **Return-path scrubber.** `_scrub` (`safe_bridge.py:255-277`)
   applies the repository's foundry-token deny-list and absolute-path
   regexes to every string field of every SKILL return value (15
   anchored return sites) before it crosses back to the agent loop.
4. **Reasoning-trace scrub.** `reasoning_content` / `thinking` is
   scrubbed at every LLM-client sink: Kimi (`llm_client.py:339`),
   MiniMax (`:431`), OpenAI (`:533`), Xiaomi MiMo (`:636`), DeepSeek
   (`:742`), and the Ollama return path / debug-log sinks (`:818`,
   `:810`) — the latter pair closed in the e750189c rework after a
   P0 was caught on re-review. Every vendor with a separate thinking
   / reasoning channel is covered by the same `scrub` wrapper
   (`safe_bridge.py:280-290`) before the trace re-enters
   conversation history.

## 3.4  Five hard invariants (summary)

Every exposed call enforces the same five invariants, so a
misconfigured `allowed_params` cannot reopen a closed channel:
(1) PDK-token redaction on every output; (2) per-project parameter-
name whitelist; (3) SKILL-entrypoint allow-list with nested-call
extension; (4) inline-upload directory containment; (5) op-point
key whitelist (`_SAFE_OP_POINT_KEYS`, `safe_bridge.py:41-56`).
Code anchors for each invariant and the full tier-by-tier threat
model are in the supplementary appendix.

## 3.5  Trust-boundary status — compressed Table

| Tier | Attack class                          | Status                         |
|------|---------------------------------------|--------------------------------|
| 1    | Naive prompt injection                | enforced                       |
| 2    | Schema / SKILL injection              | enforced                       |
| 3    | Reasoning-trace re-entry (replay)     | enforced (replay closed via scrub layer 4, §3.3) |
| 3    | Reasoning-trace first-emission        | out of scope                   |
| 4    | Timing / error-class side channel     | partial                        |

The first-emission row remains out of scope under the curious-but-
passive model — provider-side request logs capture unscrubbed
reasoning bytes on first emission, but we do not claim coverage of
near-active providers. Closing it requires a hash-only thinking
channel or fully on-prem inference (§8 future work).

## 3.6  Why §3 matters for SP2

Without §3 the benchmark cannot run on a real industrial PDK —
every LLM-bound byte would egress foundry content. Eleven
checkpoints across seven vendor families amplifies that risk by an
order of magnitude (eleven sets of provider logs instead of one).
Safe-Bridge is the substrate; SP2 (§4–§5) is what it lets us
measure.
