## 4.1  Threat actor

The adversary in this paper is **the external LLM provider** — Anthropic,
OpenAI, Moonshot (Kimi), MiniMax, or any other hosted inference endpoint
the agent calls through `src/llm_client.py`. The provider is assumed to
be:

1. **Curious-but-passive** in the cryptographic sense: it sees every
   prompt and every tool-result string we hand it, and it may log,
   train on, or otherwise mine that traffic, but it does **not** mount
   active man-in-the-middle attacks against the Cadence host.
2. **Capable of crafting tool-call arguments** that look syntactically
   well-formed but are designed to exfiltrate protected content — i.e.
   the LLM itself is the prompt-injection vector against its own
   sandbox, not an external user.

We do **not** model a malicious internal user with shell access to the
Cadence host, nor a foundry-side breach of the PDK tarball at rest.
Those are handled by OS-level permissions and out of scope (§4.5).

## 4.2  Protected asset

The asset is **foundry NDA-bound PDK content**, in three concrete forms:

- **Device model cards** — BSIM4 / BSIMCMG parameter tables shipped as
  `*.scs` / `*.l` files. Leakage of even a single `vth0`, `u0`, or
  `cgdo` value violates the foundry NDA.
- **Process layer stack** — metal-pitch / via-resistance / well-doping
  tables in the technology file.
- **Proprietary cell topologies and naming** — the foundry's standard-
  cell library names and device-family prefixes embed process-node
  identification. The exact public deny-list lives in the scrubber
  implementation, not in prose.

Concretely, the agent must never let any of those byte sequences reach
the LLM provider, either as a prompt token, a tool-result string, or a
log line that the LLM subsequently re-reads as conversation history.

## 4.3  Attack ladder

We organise observed and anticipated attacks into four tiers of
increasing subtlety. For each tier we state the defence, the code
anchor, and whether the defence is **enforced**, **partial**, or
**deferred**.

### Tier 1 — Naive prompt injection

The LLM emits a tool call whose arguments overtly request foundry
content by naming a private PDK library, a private device cell, or a
PDK model path directly.

**Defence — enforced.**

- **No `read_file` or generic filesystem tool is exposed.** The bridge
  exposes 17 specific SKILL entrypoints (`safe_bridge.py:125-176`,
  `_ALLOWED_SKILL_ENTRYPOINTS`) plus `set_instance_param` /
  `read_schematic` / `read_op_point`. Arbitrary `load()` / `hiOpenLib`
  / `dbOpenCellViewByType` are rejected at
  `_check_skill_entrypoint` (`safe_bridge.py:2139-2202`).
- **PDK-name regex on the return path.** `_FOUNDRY_LEAK_RE`
  (`safe_bridge.py:80-83`) matches foundry-shaped tokens
  case-insensitively and is applied by `_scrub` (`safe_bridge.py:255-277`)
  to every error message and every string field of every SKILL return
  value. Foundry-name tokens are replaced with `<redacted>` before the
  string is handed to the agent loop.
- **Absolute-path scrubbing.** `_ABS_WIN_PATH_RE`, `_ABS_UNIX_PATH_RE`,
  `_UNC_PATH_RE`, `_FORWARD_UNC_PATH_RE` (`safe_bridge.py:84-95`)
  collapse Windows / POSIX / UNC paths to the literal `<path>` so the
  LLM cannot reconstruct PDK directory layout from error messages.

### Tier 2 — Schema / SKILL injection

The LLM crafts a parameter that *looks* like a value but smuggles a
SKILL expression, for example by terminating a value and attempting a
private-library open call, or an entrypoint with a Unicode lookalike to
slip past the allowlist.

**Defence — enforced.**

- **Strict parameter-name whitelist.** `_is_allowed_param_name`
  (`safe_bridge.py:2300-2346`) admits only names that (a) are members
  of `self.allowed_params` (default `["w", "l", "nf", "m", "multi",
  "wf", "r", "c", "nfin", "fingers", "idc", "vdc"]`, configurable via
  the per-project YAML at construction time, line 371-381), (b) match
  `^[a-zA-Z][a-zA-Z0-9_]{0,31}$`, (c) avoid the BSIM-intrinsic blocklist,
  and (d) do not match `_FOUNDRY_LEAK_RE`.
- **Strict identifier regex on every lib/cell/instance string.**
  `_NAME_RE = ^[a-zA-Z0-9_.\-]+$` (`safe_bridge.py:34`), enforced by
  `_validate_name` (line 280). No quotes, semicolons, backticks,
  whitespace, or nested parens can survive into a SKILL expression.
- **Numeric-literal regex on parameter values.** `_PARAM_ATOM_RE`
  (`safe_bridge.py:35-37`) accepts only a single signed
  decimal/exponent atom with an optional SI-suffix; multi-statement
  payloads are rejected at the value boundary.
- **Pre-format `.scs`-path quote/backslash/semicolon filter**
  (`safe_bridge.py:1177-1182`) blocks SKILL string-literal breakout via
  the netlist-IC path.
- **ASCII-only + control-char rejection on SKILL entrypoints**
  (`safe_bridge.py:2156-2179`) closes the Unicode-lookalike and
  embedded-NUL channels.
- **Nested-call allowlist.** After matching the top-level entrypoint,
  the bridge greps the full expression for `identifier(` patterns and
  rejects any that are neither in `_ALLOWED_SKILL_ENTRYPOINTS` nor in
  `_ALLOWED_SKILL_NESTED = {"list"}` (`safe_bridge.py:2196-2202`).

### Tier 3 — Reasoning-trace sniffing

A reasoning model (Kimi K2.5, MiniMax M2.7, OpenAI o-series, etc.) is
allowed to emit a long `reasoning_content` / thinking block in addition
to its public content. If at turn *N* a PDK-tainted string leaked into
the conversation (say, through an error path we didn't anticipate), the
model may incidentally **paraphrase** that content inside its
turn-*N+1* thinking block — bypassing our content-channel scrubber if
the thinking block re-enters the conversation unfiltered, because the
thinking block is returned by the *provider*, not by SKILL.

**Defence — enforced (R2 dual APPROVE, 2026-05-12).**

- **`reasoning_content` / `thinking` is scrubbed at every LLM-client
  sink** — both the value returned to history (Kimi
  `llm_client.py:335`, MiniMax `:427`, Ollama empty-content fallback
  `:497`) and the debug-log sink (Ollama `:489`, surfaced as a P0 on
  codex_reviewer_v2's re-scan and closed in the bundled rework before
  re-review). `scrub` is the public wrapper around `_scrub`
  (`safe_bridge.py:280-290`), reusing the same `_FOUNDRY_LEAK_RE` /
  abs-path regexes that defend the Tier 1 return path. Re-entry into
  history at `agent.py:610-612` and any debug-log emission see only
  redacted bytes.
- **Regression coverage.** `tests/test_llm_client.py::TestOllamaThinkingFallback::test_debug_log_scrubs_thinking_pdk_tokens`
  injects an eight-token PDK-tainted thinking block and asserts on
  three independent axes (records-present, forbidden-tokens-absent,
  scrub-marker-present) — see §4.7 for the audit methodology this
  test embodies.
- **Residual first-emission exposure.** The provider's request log
  still captures the unscrubbed reasoning block when it is first
  emitted by the model. Under the curious-but-passive threat model
  we accept this: the provider already sees every prompt and every
  tool-result string, so the marginal information gain from one extra
  paraphrase is small. A near-active provider would warrant going
  further — see §4.5 (out of scope) and §8 (future work).
- **Why redaction over discard.** We considered the cheaper mitigation
  of discarding `reasoning_content` entirely. We chose redaction
  because reasoning-content carries the only signal that lets us
  reason about Tier 3 attacks empirically (§5 failure-mode bucket
  *reasoning-content replay*) and because reasoning tokens are billed
  even when discarded (§6 cost Pareto would systematically
  under-report Kimi / MiniMax cost).

### Tier 4 — Side channels

Even with content scrubbed perfectly, two side channels remain:

1. **Timing.** Per-iteration latency depends on PDK-corner section and
   on whether a particular device is in saturation vs. triode. A
   patient adversary fitting a model over thousands of iterations could
   in principle recover a coarse threshold-voltage histogram.
2. **Error-class side channel.** A SKILL call that fails with
   `RuntimeError` (caught, scrubbed) leaks one bit ("library X exists /
   contains cell Y / accepts parameter Z") per call, even though the
   error *string* is `<redacted>`.

**Defence — partial.**

- The scrubber removes content but does not equalise latency or
   error rates. A timing-channel mitigation (constant-rate token
   release, dummy SKILL calls) would push iteration cost up by an
   estimated 1.3-2× and is not in the submitted artefact.
- We argue that under the curious-but-passive threat model the
  surviving channel bandwidth is low enough to be acceptable for a
  research demonstrator; productionising against a near-active
  provider would require closing it. This is listed as future work.

## 4.4  Defended invariants

The bridge enforces five hard invariants. Each is checked on every
exposed call, not at construction time, so a misconfigured
`allowed_params` list cannot reopen a closed channel.

1. **PDK-token redaction on every output.** No string returned to the
   agent loop, whether a normal result, a structured warning, or an
   exception message, contains a token matching `_FOUNDRY_LEAK_RE` or
   an absolute filesystem path. `_scrub` (`safe_bridge.py:255-277`) is
   applied at every return site (lines 552, 912, 1104, 1194, 1255,
   1318, 1391, 1535, 1692, 1883, 1998, 2215, 2223, 2266) and at the
   LLM-client boundary for both the return-path and the debug-log
   sink of `reasoning_content` / `thinking`
   (`llm_client.py:335, 427, 489, 497`; regression coverage in
   `tests/test_llm_client.py`).

2. **Per-project parameter-name whitelist.** Only names in
   `allowed_params` (per-project YAML, defaulted on construction) and
   passing the regex/blocklist gate of `_is_allowed_param_name`
   (`safe_bridge.py:2300-2346`) ever reach a SKILL `setattr`. BSIM
   intrinsics (`vth0`, `u0`, `cgdo`, ...) are blocklisted explicitly.

3. **SKILL-entrypoint allowlist.** Only the 17 names in
   `_ALLOWED_SKILL_ENTRYPOINTS` (`safe_bridge.py:125-176`) are
   forwardable; the nested-call grep (lines 2196-2202) extends this
   guarantee to embedded subexpressions. There is **no** generic
   `eval_skill(expr)` tool.

4. **Inline-upload directory containment.** `_upload_skill_inline`
   (`safe_bridge.py:2021-2024`) refuses any path that fails
   `path.relative_to(self._skill_dir)`, where `_skill_dir` resolves to
   the in-repo `skill/` directory only. The agent cannot inject a SKILL
   wrapper from an arbitrary filesystem location.

5. **Operating-point key whitelist.** `_SAFE_OP_POINT_KEYS`
   (`safe_bridge.py:41-56`) admits the small fixed set of device-physics
   quantities (`vgs`, `vds`, `id`, `gm`, `gds`, `vth`, `vdsat`, `region`,
   terminal capacitances). Model-card-derived keys are stripped by
   `_strip_model_info` (`safe_bridge.py:2282-2291`) before the operating
   point ever reaches the LLM.

## 4.5  Out of scope

- **Insider threat.** A user with shell access to the Cadence host can
  read the PDK directly; SafeBridge sits in the Python process, not at
  the OS layer.
- **Direct PDK theft.** Filesystem permissions on the foundry tarball
  are the foundry's and the customer IT department's responsibility,
  not ours.
- **Supply-chain attacks on Cadence Virtuoso / IC23.1.** We assume the
  Cadence binary and the RAMIC bridge implementation
  (virtuoso-bridge-lite) are trustworthy.
- **Active MITM on the LLM API channel.** TLS to the provider is
  assumed sound; if it is not, the leak surface is far wider than
  anything SafeBridge can defend.
- **Foundry-side adversaries.** Mitigated by the foundry's own
  controls; outside our scope.
- **First-emission capture of `reasoning_content` by the provider.**
  Patch (i) redacts the *re-entry* leg; the bytes the model emits in
  the first place still hit the provider's request log. Closing that
  requires a model that exposes a hash-only thinking channel or a
  fully on-prem inference stack (Ollama, addressed orthogonally in
  §7 SP4).

## 4.7  Audit methodology — defense-in-depth + multi-axis tests

Two patterns from the Safe-Bridge implementation are worth surfacing
as reusable practice for any future closed-loop EDA agent:

1. **Type-narrowed live paths + non-string fall-through as
   defense-in-depth.** All live callers of `scrub` pre-narrow the
   argument to `str` via `(getattr(msg, ..., None) or "").strip()`
   (`llm_client.py:332-333, 424-425, 485-486`), so the regex pass
   inside `_scrub` always sees a string. The fall-through that
   returns non-string inputs unchanged (`safe_bridge.py:265-268`) is
   therefore *not* on a live path; it is a safety net that prevents
   a future caller from accidentally creating a leak channel by
   passing a wrapped type. R2 reviewers (`claude_reviewer_v2`,
   `codex_reviewer_v2`) confirmed both halves of this invariant.

2. **Multi-axis scrub regression tests.** A single negative assertion
   ("the forbidden token is absent from the log") gives a false
   positive when the log line itself is never emitted. Our scrub
   regressions assert on three independent axes:
   (a) **records-present** — the expected log record is emitted at
   the expected level;
   (b) **forbidden-tokens-absent** — every member of a known PDK-
   token set is absent from the emitted text;
   (c) **scrub-marker-present** — the redaction marker (`<redacted>`
   / `<path>`) is present, confirming the scrubber ran rather than
   the leak path being unreachable in the test fixture.
   `test_debug_log_scrubs_thinking_pdk_tokens`
   (`tests/test_llm_client.py:134-178`) is the canonical example;
   the same three-axis pattern is reusable for any return-path or
   log-sink scrub.

These two patterns are paper-level audit observations, not new code;
we surface them because reviewers asked "how do we know the scrub
ran?" — and "we wrote a regression that fails three different ways
if it didn't" is a more satisfying answer than "we read the code".
